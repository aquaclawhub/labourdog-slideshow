做一笔劳务支付， 为什么总感觉流程很长？

因为它不是一个动作，而是一串动作。

从最开始收集人员信息开始，一笔劳务通常要经过：

钉钉劳务人员录入 → 钉钉任务下达 → 通知劳务管理员确认 → 税务确认 → 用友表头核对 → 用友多张表格导出 → 签署文件/国外信函处理

不是某一步特别复杂， 而是整条链条都在消耗注意力。

真正消耗时间的， 往往不是判断，而是重复。

同一批劳务信息会不断重复出现。

同一个姓名、证件号、银行账号、项目名称、核算月份，会被不同系统和不同模板反复使用。

初始 Excel 来源不统一，表头命名不一致，一个批次可能有多个人。

国内居民和国外非居民规则不同，人民币、外币、税前、税后口径并存。

流程没有被组织起来。 信息也没有。

3

劳务狗狗要解决的， 不是某一个页面，而是这条流程。

4

先有一个批次， 再有这一批人。

劳务狗狗把这件事拆成两个核心对象：支付批次与劳务人员。

这样拆开之后，很多原来混在一起的动作就有了明确边界。

5

第一步，不是去填系统， 而是先把信息收拢。

在劳务狗狗里，真正的起点是 S 一次性录入。

这个页面是整套流程的主数据页，用来统一维护：批次头信息、人员主信息、人员结算信息、税务与导出信息。

支持手工录入和 Excel 导入，后面的钉钉录入、税务确认、用友导出都统一从这里读取，不再各自重新整理。

10

有些计算， 不应该继续靠手工来回试。

国内居民人民币场景：税前正推税后 / 税后反推税前 / 手动模式

联动相关金额字段：税前金额、税后金额、个税、不含税金额、价税合计、付款金额

国外非居民和外币场景保留人工调整权，不强行做不稳定自动化。

该自动的自动， 该人工的人工。

11

有些推进动作，不应该卡在「我该怎么说」上。

系统按当前批次自动生成发给劳务管理员的通知语。

文案自动带上关键要素：项目方、项目名称、核算年月、人数、币种。

12

到了财务确认这一步， 不应该再从零整理一次。

SQ 页面提供批次级税务确认表能力。

基于当前批次人员数据生成：批次摘要、表格预览、一键导出 Excel。

导出文件遵循固定列顺序、固定表头行、固定数据起始行规则。

13

最靠后的环节，往往最不应该再耗费注意力。

表头复制/核对区

关键信息卡片，便于逐项复制。

报销明细导出区

预览 + 一键导出 Excel。

结算信息导出区

预览 + 一键导出 Excel。

工作量明细导出区

按模板生成工作量明细表。

14

有些流程尾部的小环节， 单独看不大，合起来很耗。

15

有些工具， 越靠近使用现场，越有效。

本地运行 · 本地保存 · 离线可用

不依赖真实钉钉 API · 不依赖真实用友 API

没有登录、没有后端、没有多用户系统

优先解决：项目经理每天真实会遇到的高频、重复、容易出错的流程段落

16

劳务狗狗真正改变的， 不是某一张表，而是整个过程里的节奏。

过去

收表 → 整理 → 复制 → 再整理 → 再复制 → 再核对 → 再导出

现在

录入 → 确认 → 输出 → 推进

结果

少一次重复，就少一次出错。 少一次出错，就少一次返工。
